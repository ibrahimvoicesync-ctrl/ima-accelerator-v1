  implication: Non-RFC UUIDs like 00000000-0000-0000-0000-000000000002 fail validation

## Resolution

root_cause: Zod v4 (4.3.6) enforces RFC 9562/4122 UUID format (version nibble [1-8], variant nibble [89abAB]). The seed data in supabase/seed.sql uses deterministic UUIDs like '00000000-0000-0000-0000-000000000002' that have version=0 and variant=0, which are not RFC-compliant. Ibrahim Awwad's UUID works because it was generated by Postgres gen_random_uuid() producing a real v4 UUID. Two interacting causes: (1) Zod upgraded from v3 (permissive regex) to v4 (strict regex), and (2) seed data uses fake non-RFC UUIDs.
fix: Two-part fix needed: (1) Change z.string().uuid() to z.string().regex(/^[0-9a-fA-F]{8}-...$/) or use z.string().guid() which is the permissive 8-4-4-4-12 pattern in Zod v4. OR (2) Fix seed data to use RFC-compliant v4 UUIDs. Recommended: use z.string().guid() in API validation since it accepts any 8-4-4-4-12 hex pattern, and also fix seed UUIDs for correctness.
