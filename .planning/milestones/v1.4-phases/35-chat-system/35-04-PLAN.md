**Key difference from coach page (D-05):** Student chat is a SINGLE thread view — no conversation list needed because students only talk to their assigned coach. The page is simpler: header bar + MessageThread + ChatComposer.

- `messages: MessageWithSender[]` — all messages in the conversation with their coach
- `hasMore: boolean` — whether older messages exist for pagination
- `isLoadingMore: boolean` — pagination loading state
- `isLoading: boolean` — initial load state

- `routerRef = useRef(useRouter())` — stable router reference

Initialization (useEffect on mount):
1. Get auth user: `const supabase = createClient(); const { data: { user } } = await supabase.auth.getUser()`
2. Fetch profile: GET from a lightweight source or use admin-fetched data. Since this is a client component, fetch profile via the messages API itself (the GET response includes profile info) or add a dedicated profile fetch.
   - Simplest approach: fetch `const res = await fetch("/api/messages"); if (!res.ok) throw new Error(...)` — the API already validates auth and returns messages. Extract `coachId` from the first message's `coach_id` field, or better: make a separate call to get the user profile.
   - Better approach: Use Supabase client to get user metadata, then use auth_id to determine coach_id. Since the student's coach_id is needed BEFORE any message fetch, and we can't use admin client on the client side:
     - Fetch user profile via the Supabase client: `supabase.from("users").select("id, role, coach_id, name").eq("auth_id", user.id).single()` — this works because RLS on users table allows self-read.
     - Actually, check RLS: the student can read their own user row. If not, the GET /api/messages endpoint already returns messages correctly filtered by coach_id. So: call GET /api/messages on mount — if it returns messages, extract coach_id from messages[0].coach_id. If no messages, the student has no conversation yet.
     - Best approach: Modify the init to also fetch from `/api/messages?type=init` or just use the Supabase client auth user. The simplest pattern matching the codebase: use the `createClient()` from `@/lib/supabase/client`, call `getUser()`, then use the auth user's ID for all API calls. The API routes handle coach_id resolution server-side.
   - **Final approach:** On mount, call `const profileRes = await fetch("/api/messages?type=profile")`. But wait, our GET /api/messages doesn't have a profile mode. Instead:
     - Just start polling. The GET /api/messages endpoint already filters by the student's coach. On first poll, set `currentUserId` from the Supabase client auth, and `coachId` will be inferred from messages. But we need coachId for mark-as-read.
     - **Simplest working approach:** The page reads auth from Supabase client on mount to get `user.id`. Then it needs `profile.id` and `profile.coach_id`. Since client components can't use admin client, and the users table RLS allows students to read their own row, use: `const { data: profile } = await supabase.from("users").select("id, coach_id").eq("auth_id", user.id).single()`. Set `currentUserId = profile.id`, `coachId = profile.coach_id`.
3. If `coachId` is null: show an error/empty state — "No coach assigned. Contact your administrator." Do NOT start polling.
4. If `coachId` exists: mark messages as read immediately: `fetch("/api/messages/read", { method: "PATCH", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ coach_id: coachId }) })` — check response.ok
5. Set `isLoading = false`, start polling
