# Phase 63-01: DIY Owner Detail Page (F6)

<objective>
Make `/owner/students/[studentId]` work for `student_diy` users by broadening role filters on the owner detail route + owner list, add a "DIY" badge on the list, and suppress daily-report-derived UI inside CalendarTab and StudentKpiSummary for DIY students via a `role` prop threaded through existing components. No parallel route tree, no per-role sub-components, no migrations.
</objective>

<must_haves>
1. Navigating to `/owner/students/{diyStudentId}` renders the detail page with no 404/500 (DIY-01, DIY-10)
2. `/owner/students` lists DIY users with a "DIY" Badge, layout unchanged at 375px (DIY-02, DIY-03)
3. `role: "student" | "student_diy"` prop is threaded from server through `OwnerStudentDetailClient` → `StudentDetailTabs`/`CalendarTab`/`StudentKpiSummary` (DIY-04)
4. `CalendarTab` suppresses daily-report dots and the "Daily Report" panel card when `role === "student_diy"`; session activity still renders (DIY-05, DIY-06)
5. `StudentKpiSummary` suppresses report-derived rows (Daily Outreach, Hours Worked, Lifetime Outreach sourced from reports) for DIY users — keeps step/non-report content; or hides the whole card if all remaining content is report-derived (DIY-05)
6. Regular `role="student"` behavior is functionally identical to v1.7 (DIY-07)
7. `/coach/students/[studentId]` is NOT modified (DIY-08)
8. No parallel `/owner/students_diy/...` route tree is created; no `OwnerStudentDetailClientDIY.tsx` file is added (DIY-09)
9. `get_student_detail` RPC is role-agnostic (takes `p_student_id` only) — no migration required (DIY-10, confirmed via audit of `supabase/migrations/00011_write_path.sql:292`)
10. Post-phase build gate: `npm run lint && npx tsc --noEmit && npm run build` exits 0
</must_haves>
