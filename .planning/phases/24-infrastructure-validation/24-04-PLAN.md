    - load-tests/scenarios/combined.js (lines 13-15, 119 — same SharedArray pattern)

  <action>
In load-tests/scripts/gen-tokens.js, change line 168 from:

```javascript
fs.writeFileSync(ownerTokenPath,    JSON.stringify(ownerToken,    null, 0));
```

to:

```javascript
fs.writeFileSync(ownerTokenPath,    JSON.stringify([ownerToken],  null, 0));
```

This wraps the ownerToken string in an array before stringifying. The result file will contain `["eyJ..."]` instead of `"eyJ..."`.

Why: read-mix.js (line 9-10) and combined.js (line 13-15) both use `new SharedArray('owner-token', function() { return JSON.parse(open('../tokens/owner_token.json')); })`. SharedArray requires the init function to return an array. When the file contains a plain JSON string, JSON.parse returns a string, and SharedArray wraps it — but accessing `[0]` on a string returns its first character ('e'), not the full JWT. Wrapping in an array makes JSON.parse return `["eyJ..."]`, and `[0]` correctly returns the full token.

Do NOT modify read-mix.js or combined.js — they are already written correctly for array consumption. The bug is solely in the writer (gen-tokens.js).
  </action>
