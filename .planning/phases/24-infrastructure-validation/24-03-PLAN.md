<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Run full load test scenarios and capture connection data</name>
  <files>load-tests/results/ (generated, not committed)</files>
  <action>
    Human runs all three k6 scenarios and captures monitoring data. Claude cannot do this because it requires active network access to the staging environment and manual Supabase dashboard monitoring (D-16).

    **Important timing note (Pitfall 6):** Run tests between 10 AM - 12 PM UTC to avoid pg_cron jobs at 2 AM and 3:30 AM UTC.

    **Scenario 1 — Read Mix:**
    ```bash
    "/c/Program Files/k6/k6.exe" run --summary-trend-stats="med,p(95),p(99)" --out json=load-tests/results/read-mix-results.json -e APP_URL=<staging-app-url> -e SUPABASE_URL=<staging-supabase-url> -e SUPABASE_ANON_KEY=<staging-anon-key> load-tests/scenarios/read-mix.js
    ```
    Record from terminal output: P50, P95, P99 latency, error rate, peak VUs.

    **Scenario 2 — Write Spike:**
    ```bash
    "/c/Program Files/k6/k6.exe" run --summary-trend-stats="med,p(95),p(99)" --out json=load-tests/results/write-spike-results.json -e APP_URL=<staging-app-url> load-tests/scenarios/write-spike.js
    ```
    **During the sustained peak (minutes 2-7)**, capture connection counts from Supabase dashboard (D-16) or run this SQL:
    ```sql
    SELECT count(*) AS total_connections, count(*) FILTER (WHERE state = 'active') AS active, count(*) FILTER (WHERE state = 'idle') AS idle, (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') AS max_connections, round(count(*) * 100.0 / (SELECT setting::int FROM pg_settings WHERE name = 'max_connections'), 1) AS pct_used FROM pg_stat_activity WHERE datname = current_database();
    ```

    **Scenario 3 — Combined:**
    ```bash
    "/c/Program Files/k6/k6.exe" run --summary-trend-stats="med,p(95),p(99)" --out json=load-tests/results/combined-results.json -e APP_URL=<staging-app-url> -e SUPABASE_URL=<staging-supabase-url> -e SUPABASE_ANON_KEY=<staging-anon-key> load-tests/scenarios/combined.js
    ```

    **After all tests**, capture rate limiter data (D-17):
    ```sql
    SELECT endpoint, count(*) AS total_calls, count(DISTINCT user_id) AS unique_users FROM rate_limit_log WHERE called_at > now() - interval '60 minutes' GROUP BY endpoint ORDER BY total_calls DESC;
    ```

    **Capture pg_stat_statements top queries:**
    ```sql
    SELECT query, calls, round(mean_exec_time::numeric, 2) AS mean_ms, rows FROM pg_stat_statements WHERE query NOT LIKE '%pg_stat%' ORDER BY mean_exec_time DESC LIMIT 10;
    ```
  </action>
  <verify>Human provides: all P50/P95/P99 numbers for 3 scenarios, connection counts during spike, rate_limit_log results, pg_stat_statements top 10, and any 429 response counts.</verify>
  <done>All three k6 scenarios executed. P50/P95/P99 latency, error rates, connection counts, rate limiter trigger counts, and slow query data captured and reported.</done>
</task>
