    **Seed SQL script (`load-tests/seed/00001_staging_seed.sql`):**

    Use set-based `INSERT INTO ... SELECT FROM generate_series()` pattern (NOT PL/pgSQL FOR LOOP per Pitfall 4 — a loop over 500k rows takes 20-40 min and hits Supabase SQL editor timeout).

    The seed must create:
    1. **1 owner** — email: `owner@test.ima`, name: `Test Owner`, role: `owner`, status: `active`
    2. **10 coaches** — email: `coach{N}@test.ima` (N=1..10), role: `coach`, status: `active`
    3. **5,000 students** — email: `student{N}@test.ima` (N=1..5000), role: `student`, status: `active`, coach_id randomly assigned to one of the 10 coaches

    For each student, generate:
    4. **~90 daily_reports** — one per day for 90 days back from CURRENT_DATE. Columns:
       - `student_id`: the student's UUID
       - `date`: each date in the 90-day range
       - `hours_worked`: random between 2.0 and 8.0 (decimal)
       - `star_rating`: random 1-5
       - `outreach_count`: random 0-15
       - `brands_contacted`: random 0-10
       - `influencers_contacted`: random 0-5
       - `calls_joined`: random 0-3
       - `submitted_at`: `(date + make_interval(hours => submission_hour))::timestamptz` where submission_hour is 21 or 22 for 80% of rows (D-07: 9-11 PM UTC spike), and random 0-20 for the other 20%

    5. **~2-3 work_sessions per student per day** (for first 30 of 90 days, keeping volume reasonable ~300k rows). Columns:
       - `student_id`, `date`, `cycle_number`: 1, 2, or 3
       - `session_minutes`: randomly 30, 45, or 60
       - `started_at`: date + random hour offset
       - `completed_at`: started_at + session_minutes interval
       - `duration_minutes`: same as session_minutes
       - `status`: `completed`

    6. **10 roadmap_progress rows per student** (steps 1-10). First 3 steps `completed`, step 4 `active`, steps 5-10 `locked`. `step_name` values: `Join Course`, `Watch Intro`, `Set Up IG`, `First DM`, `Book Call`, `Sign Client`, `First Invoice`, `Testimonial`, `Scale Up`, `Brand Deal`.

    **Critical: auth_id generation strategy.**
    The seed script must generate deterministic UUIDs for the `auth_id` column so the JWT pre-gen script can mint tokens with matching `sub` claims. Use a pattern like:
    ```sql
    -- Students: auth_id derived from index
    -- Format: 00000000-0000-4000-a000-{12-digit zero-padded index}
    -- Example student 1:  00000000-0000-4000-a000-000000000001
    -- Example student 5000: 00000000-0000-4000-a000-000000005000
    ```
    This deterministic pattern lets the JWT script generate matching UUIDs without querying the DB.

    The owner and coaches also get deterministic auth_ids:
    - Owner: `00000000-0000-4000-b000-000000000001`
    - Coach N: `00000000-0000-4000-b000-0000000000{N+1 zero-padded to 12}` (e.g., coach 1 = ...000000000002)

    **Performance approach:** Wrap all INSERTs in a single transaction. Use CTEs or temp tables to hold generated student UUIDs, then cross-join with generate_series(0,89) for daily_reports. The script should execute in under 5 minutes for 500k+ rows.

    Add a comment header explaining: "Run via `supabase db execute --sql-file load-tests/seed/00001_staging_seed.sql` against the staging project. Do NOT run against production."

    **JWT pre-generation script (`load-tests/scripts/gen-tokens.js`):**

    A Node.js CommonJS script that:
    1. Reads `STAGING_JWT_SECRET` from env var (exit with error if missing)
    2. Generates 5,000 student auth_id UUIDs using the same deterministic pattern as the seed SQL (00000000-0000-4000-a000-{12-digit padded index})
    3. Also generates 1 owner + 10 coach auth_ids using the b000 pattern
    4. For each UUID, mints a JWT with:
       - `sub`: the auth_id UUID
       - `role`: `authenticated`
       - `aud`: `authenticated`
       - `iss`: `supabase`
       - `exp`: 7 days from now (Unix timestamp in seconds)
    5. Writes three files:
       - `load-tests/tokens/student_tokens.json` — array of 5,000 student JWT strings
       - `load-tests/tokens/owner_token.json` — single owner JWT string (for read-mix scenario)
       - `load-tests/tokens/student_uuids.json` — array of 5,000 student auth_id UUIDs (for reference)
    6. Uses `require('jsonwebtoken')` — the `jsonwebtoken` npm package (install: `npm install jsonwebtoken`)
    7. Prints summary: "Generated {N} student tokens, 1 owner token, 10 coach tokens"

    Usage: `STAGING_JWT_SECRET=<secret> node load-tests/scripts/gen-tokens.js`

    Create `load-tests/tokens/.gitkeep` as an empty file so the directory is tracked but token files are not.
