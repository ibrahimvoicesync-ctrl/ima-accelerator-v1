# Phase 59: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Reviewed the Phase 59 Referral API route (`POST /api/referral-link`) and its companion smoke runner. The route is well-structured and meets the v1.7 idempotency contract: CSRF -> Auth -> Role -> Env -> Body -> Cache-hit short-circuit -> code generation -> Rebrandly fetch (with timeout + ok check + try/catch) -> compare-and-swap persist -> success. All core security requirements pass:

- Auth and role gate execute BEFORE body parse, code generation, or Rebrandly call.
- Admin Supabase client is used for every `.from()` query (Hard Rule 4).
- `response.ok` is checked before parsing Rebrandly JSON (Hard Rule 6).
- Zod is imported as `from "zod"` (Hard Rule 7).
- `https://` scheme is prepended BEFORE persist (Pitfall 2 / prior_decisions Q5).
- Compare-and-swap UPDATE (`.is("referral_short_url", null)`) prevents overwrite on race (API-05 Pattern 2).
- `REBRANDLY_API_KEY` is never logged or returned in any response body.
- Every `catch` calls `console.error` (Hard Rule 5 — no swallowed errors).
- Timeout (8s) on Rebrandly fetch via `AbortSignal.timeout(8000)`.

The smoke runner correctly skips cases that require runtime artifacts (session cookies, env flips) and exits non-zero on failure.

Two **warnings** worth noting:
1. The runner's `.env.local` parser is overly permissive — it strips wrapping quotes greedily and uses a regex that does not handle `=` inside values, which could mis-parse a Rebrandly key containing `=` padding. Minor, but worth tightening.
2. SMOKE 6 mutates production-shaped data (`update referral_short_url = null` on a real student row) without a transaction or restore step — if the runner crashes between reset and POST #1, the test student is left with a wiped short URL.

The remaining **info** items are minor: a UUID-based code generator with no collision retry, a small dead branch in the post-CAS re-read, and a few opportunities to harden the smoke runner's auth-credential handling.

## Warnings

### WR-01: Smoke runner `.env.local` parser strips trailing quotes too greedily

**File:** `scripts/phase-59-smoke-runner.cjs:36-39`
**Issue:** The parser uses `.replace(/^['"]|['"]$/g, "")` which strips a leading OR trailing single/double quote independently. A value like `foo"bar` (no surrounding quotes, but containing one) loses its trailing `"`. More importantly, the regex `/^([A-Z_][A-Z0-9_]*)=(.*)$/` greedy-matches the FIRST `=` only via the capture group, which is correct — but if a value spans newlines or has CRLF artifacts, the parser silently drops it. Since the runner is the source of truth for service-role credential loading, parser quirks risk silent auth failures that look like infra problems.
**Fix:** Either delegate to `dotenv` (already in the dep tree if `@supabase/supabase-js` is present, or trivial to add as a devDependency) OR tighten the strip to only remove a matched pair:
```js
let value = m[2];
if ((value.startsWith('"') && value.endsWith('"')) ||
    (value.startsWith("'") && value.endsWith("'"))) {
  value = value.slice(1, -1);
}
env[m[1]] = value;
```

### WR-02: SMOKE 6 wipes a real student's `referral_short_url` without restore-on-failure

**File:** `scripts/phase-59-smoke-runner.cjs:257-261`
**Issue:** The runner does `update referral_short_url = null` on the `TEST_STUDENT_EMAIL` row to force the fresh-create branch. If the runner is killed (Ctrl+C, network error, Rebrandly outage between reset and POST #1, or any thrown error before POST #2 succeeds), the test student is left with a NULL short URL — meaning the next time they hit `/dashboard`, they get a NEW Rebrandly link minted, breaking the "at most one Rebrandly call per user for life" invariant for that user.

This is benign for a dedicated synthetic test student but dangerous if `TEST_STUDENT_EMAIL` is ever pointed at a real student row.

**Fix:** Snapshot the original `referral_short_url` before the reset and restore it in a `finally` block, OR document explicitly that `TEST_STUDENT_EMAIL` MUST be a dedicated synthetic account that nobody else uses. Recommended snippet:
```js
const originalShortUrl = studentRow.referral_short_url;
try {
  // reset, POST #1, POST #2, record(...)
} finally {
  if (originalShortUrl !== null) {
    await sb.from("users")
      .update({ referral_short_url: originalShortUrl })
      .eq("id", studentRow.id);
  }
}
```

## Info

### IN-01: `crypto.randomUUID()` collision: code is silently regenerated by re-read but no retry

**File:** `src/app/api/referral-link/route.ts:71-89`
**Issue:** The code generator slices the first 8 hex chars of a UUID v4. With 16^8 = 4.3B possible codes, collision probability for the project's expected user count (<10k) is negligible. The code correctly uses `.is("referral_code", null)` to skip the update if a concurrent writer beat us, and re-reads the row afterwards. However, if `codeError` fires due to a unique-constraint violation (23505) on a brand-new generated code that happens to collide with another user's code, the route returns 500 with no retry — the user is permanently blocked from generating a referral link until manual intervention.

The PLAN's prior_decisions Q3 explicitly chose "do NOT retry on 23505," so this is intentional. Worth flagging only because the failure mode is invisible to the user (they see "Failed to generate referral code" with no recovery path).
**Fix:** Either accept as-is (current decision per Q3) OR add a single-shot retry with a fresh code on 23505:
```js
if (codeError?.code === "23505") {
  // one retry with fresh code
}
```
No action required if the team is comfortable with the documented decision.

### IN-02: Dead branch — post-CAS re-read returns "Failed to save referral link" but `referralCode` is also lost

**File:** `src/app/api/referral-link/route.ts:141-156`
**Issue:** The "lost CAS but no winner found" branch logs and returns 500. This is genuinely unreachable in normal operation (CAS lost means another writer won, so a row must exist), but if it ever fires (e.g., the row was deleted between CAS and re-read), the response gives no clue to the user. Not a functional defect — just a code path that, if hit, indicates a deeper data-integrity issue (user row vanished mid-request).
**Fix:** Either accept (genuinely unreachable) OR upgrade the `console.error` to include `profile.id` so on-call can locate the affected user immediately:
```js
console.error("[POST /api/referral-link] Lost CAS but no winner found for user.id=", profile.id);
```

### IN-03: Body parse swallows JSON error silently without logging

**File:** `src/app/api/referral-link/route.ts:50-54`
**Issue:** `try { body = await request.json() } catch { body = {} }` is intentional per Pitfall 8 (empty body is valid). However, this differs from the Hard Rule 5 spirit — every catch should `console.error`. In this case the swallow is by design, but a one-line comment trace (e.g., `console.debug`) would help debug future cases where a real client sends malformed JSON and gets silently coerced to `{}`.
**Fix:** Optional. If kept, the existing line 8 comment ("Empty-body POST is valid per Pitfall 8") is sufficient justification — no action required.

### IN-04: Rebrandly title leaks user's display name to a third-party API

**File:** `src/app/api/referral-link/route.ts:99`
**Issue:** `title: \`IMA Referral - ${profile.name ?? referralCode}\`` sends the student's display name to Rebrandly. This is not a security flaw (titles are intended metadata), but it is a privacy data-flow worth documenting. If the project has a privacy-policy clause restricting PII export to third parties, the team may prefer to use `referralCode` only.
**Fix:** Optional. If the privacy policy permits, leave as-is (the name is genuinely useful in the Rebrandly dashboard). Otherwise:
```js
title: `IMA Referral - ${referralCode}`,
```

### IN-05: Smoke runner mixes `process.env` and `.env` lookups inconsistently

**File:** `scripts/phase-59-smoke-runner.cjs:41-52`
**Issue:** Lines 41-42 use `env.X || process.env.X` (file wins), but lines 49-52 use `process.env.X || env.X` (process wins). This is a minor footgun — running with both `.env.local` and a shell env var set yields different precedence depending on which key. Not a bug, but consistency aids debuggability.
**Fix:** Pick one precedence order and apply uniformly. `.env.local` wins is conventional for dev tooling; `process.env` wins is conventional for CI overrides. The smoke runner is dev-only, so `.env.local` wins is the safer default:
```js
const BASE_URL = env.SMOKE_BASE_URL || process.env.SMOKE_BASE_URL || "http://localhost:3000";
const TEST_STUDENT_COOKIE = env.TEST_STUDENT_COOKIE || process.env.TEST_STUDENT_COOKIE || null;
// etc.
```
