<objective>
Wire `revalidateTag("owner-analytics", "default")` into the four mutation routes that change the inputs of the owner-analytics RPC: POST `/api/deals`, PATCH `/api/deals/[id]`, DELETE `/api/deals/[id]`, and PATCH `/api/work-sessions/[id]` (on `completed` transition). Without this wiring, the 60s `unstable_cache` from Plan 02 hides every mutation from the owner for up to 60 seconds — the exact failure mode the v1.5 Phase 53 postmortem flagged.

Purpose: Satisfy OA-05 (owner-analytics tag invalidated on every deals mutation and on work-session completion) and PERF-04 (cache-tag invalidation wired to mutation handlers). Implement CONTEXT.md §D-04 at the scope matching the current codebase.

**Scope discrepancy between D-04 and current code (flagged for the executor):** CONTEXT.md §D-04 describes an "expanded" invalidation scope covering (1) POST `/api/work-sessions` when body.status === "completed", (2) PATCH on "minutes edit of already-completed row", and (3) DELETE of a completed work-session. **None of those code paths exist in the current repository** — POST always inserts with `status: "in_progress"` (line 173 of `src/app/api/work-sessions/route.ts`), the PATCH handler has no minutes-edit branch, and there is no standalone DELETE endpoint for `/api/work-sessions/[id]`. Therefore this plan implements the D-04 "spec minimum" that maps onto existing code paths (the four routes listed above). The expanded-scope D-04 items become relevant only if/when those endpoints are introduced; they are recorded as a deferred invariant in the summary, NOT implemented speculatively here (scope-reduction prohibition does not apply because the underlying code doesn't exist yet — implementing a handler for a nonexistent endpoint is outside this phase's scope).

Output: Three modified API route files with `ownerAnalyticsTag` imports and `revalidateTag` calls added alongside the existing tag fan-outs. All existing tag calls are preserved. No new API routes, no database changes, no UI changes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mutation handler → cache layer | After a successful DB write, the handler must fan out cache-tag invalidations. A missed fan-out leaves stale data visible for up to 60s — OA-05 violation. |
| revalidateTag failure → HTTP response | `revalidateTag` is synchronous and can throw (e.g., if the cache layer is unavailable). A failure must NOT cascade into a 500 for the user whose write succeeded. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-09 | Repudiation / Stale Data | Owner sees incorrect leaderboards after a deal mutation for up to 60s | mitigate | This plan adds `revalidateTag(ownerAnalyticsTag(), "default")` to all four relevant mutation paths. The failure mode this prevents is the exact v1.5 Phase 53 postmortem trigger referenced in 54-CONTEXT.md §canonical_refs. |
| T-54-10 | DoS (cache poisoning) | Each mutation forces all owner-analytics cache entries to re-fetch within the next 60s | accept | The owner tag is GLOBAL with a single cache entry (`["owner-analytics"]`). Each mutation costs one Postgres RPC round-trip on the next page load; at realistic mutation rates (< 10/minute) this is trivial. The alternative (skip invalidation, let 60s TTL expire naturally) would reintroduce OA-05 failure. |
| T-54-11 | Availability | `revalidateTag` throws, handler returns 500, user sees error despite successful write | mitigate | Each new `revalidateTag` call is wrapped in try/catch that logs via `console.error` and does not rethrow — same pattern as every existing tag fan-out in these routes. |
| T-54-12 | Tampering (scope drift) | D-04's "expanded" scope (POST completed, PATCH minutes-edit, DELETE completed work-session) not implemented because the code paths don't exist | accept | Documented in the plan objective. Implementing invalidation for nonexistent endpoints is speculative and would rot if/when those endpoints ARE added differently. Recorded as a deferred invariant in 54-04-SUMMARY.md so a future phase introducing those endpoints can't accidentally skip the tag. |
</threat_model>
