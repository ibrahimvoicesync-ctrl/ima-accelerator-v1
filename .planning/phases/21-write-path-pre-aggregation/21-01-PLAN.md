**Section 1: student_kpi_summaries table (per D-01)**
```sql
CREATE TABLE IF NOT EXISTS public.student_kpi_summaries (
  student_id               uuid PRIMARY KEY REFERENCES public.users(id) ON DELETE CASCADE,
  total_brands_contacted   integer      NOT NULL DEFAULT 0,
  total_influencers_contacted integer   NOT NULL DEFAULT 0,
  total_hours_worked       numeric(8,2) NOT NULL DEFAULT 0,
  total_calls_joined       integer      NOT NULL DEFAULT 0,
  total_reports            integer      NOT NULL DEFAULT 0,
  last_active_date         date,
  current_streak           integer      NOT NULL DEFAULT 0,
  last_report_date         date,
  updated_at               timestamptz  NOT NULL DEFAULT now()
);
ALTER TABLE public.student_kpi_summaries ENABLE ROW LEVEL SECURITY;
COMMENT ON TABLE public.student_kpi_summaries IS
  'Pre-aggregated KPI summaries per student. Refreshed nightly at 2 AM UTC (6 AM UAE) by pg_cron job ''refresh-student-kpi-summaries''. Advisory lock key: 2100210021.';
```
No RLS policies needed — the table is read exclusively by SECURITY DEFINER RPC functions via service_role.

**Section 2: refresh_student_kpi_summaries() function (per D-04, D-05, D-06)**
- `VOLATILE SECURITY DEFINER SET search_path = public` (VOLATILE because it writes, not STABLE)
- Advisory lock: `pg_try_advisory_lock(2100210021)` at the top. If lock not acquired, RAISE NOTICE and RETURN (skip run). Release lock with `pg_advisory_unlock(2100210021)` in both success path and EXCEPTION handler.
- Loop over all active students: `SELECT u.id AS student_id, s.last_report_date AS existing_last_report FROM users u LEFT JOIN student_kpi_summaries s ON s.student_id = u.id WHERE u.role = 'student' AND u.status = 'active'`
- **Incremental skip logic (per D-04):** For each student, check if `existing_last_report IS NOT NULL` AND no rows exist in `daily_reports` with `date > existing_last_report` and `submitted_at IS NOT NULL`. If so, CONTINUE to skip that student. This avoids re-aggregating students with no new activity since the last run. Use `last_report_date` as the sentinel for this check.
- For students NOT skipped, compute aggregates from daily_reports with `submitted_at IS NOT NULL`:
  - `COALESCE(SUM(brands_contacted), 0)` -> total_brands_contacted
  - `COALESCE(SUM(influencers_contacted), 0)` -> total_influencers_contacted
  - `COALESCE(SUM(hours_worked), 0)` -> total_hours_worked
  - `COALESCE(SUM(calls_joined), 0)` -> total_calls_joined
  - `COUNT(*)` -> total_reports
  - `MAX(date)` -> last_active_date (also used as last_report_date sentinel)
  - Per D-04: When processing, always re-SUM full lifetime for correctness (not delta arithmetic). The `last_report_date` sentinel only controls whether to skip or re-process.
- Streak computation per D-02: Use a PL/pgSQL date walk approach — iterate dates backward from MAX(date) counting consecutive calendar days with a daily_reports row (submitted_at IS NOT NULL). When a gap > 1 day is found, stop. The resulting count is current_streak. If MAX(date) < CURRENT_DATE - 1, streak is 0 (gap already exists). This is simpler than window functions and the advisory lock prevents concurrent concerns.
- Idempotent upsert: `INSERT INTO student_kpi_summaries (...) VALUES (...) ON CONFLICT (student_id) DO UPDATE SET total_brands_contacted = EXCLUDED.total_brands_contacted, ...` for all columns including `updated_at = now()`.

**Section 3: pg_cron job registration (per D-06)**
