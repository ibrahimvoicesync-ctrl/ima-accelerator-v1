## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /api/referral-link | Student clicks Get My Link; fetch crosses from client JS into the authenticated Next.js route handler (same-origin). |
| client → clipboard | navigator.clipboard.writeText carries the (server-supplied) shortUrl into the OS clipboard. |
| client → Web Share sheet | navigator.share passes shortUrl + title into the OS-level share UI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-60-01 | Tampering (XSS via shortUrl) | ReferralCard.tsx URL display | mitigate | Render shortUrl via `{shortUrl}` JSX binding only (React auto-escapes). Never use `dangerouslySetInnerHTML`. shortUrl originates from server-owned DB column, not user input — double defence. |
| T-60-02 | Spoofing (CSRF on POST) | fetch('/api/referral-link') | mitigate | API route (Phase 59) already calls `verifyOrigin(request)`. Client fetch sends same-origin with `Content-Type: application/json` which passes the check automatically. Component adds no CSRF-specific headers (none needed). |
| T-60-03 | Information Disclosure (error toast leaking server detail) | Error-path toast copy | mitigate | Toasts show only generic user-facing strings ("Could not generate your link", "Please try again.") — never expose raw error.message or server 500 body. Detailed error goes to console.error only, which is dev-visible and not sent to users. |
| T-60-04 | Denial of Service (infinite retry on transient failure) | handleGetLink | accept | User has to manually click Get My Link again on each failure — no auto-retry loop exists. API is idempotent (Phase 59 returns cached value after first success) so repeated manual clicks are safe and bounded by user action. Low-value target. |
| T-60-05 | Repudiation (missing audit trail for link generation) | API layer (out of Phase 60 scope) | accept | Phase 60 is pure UI. Phase 59 route handler already uses `console.error` on failure paths and DB persists `referral_short_url` as a permanent record. No further audit needed for the UI click. |
| T-60-06 | Tampering (navigator.share open-redirect-style abuse) | handleShare | mitigate | The URL passed to navigator.share is the server-controlled shortUrl from our DB — no user-supplied path component, no interpolation of form input. Cannot be weaponized by the user against themselves. |
